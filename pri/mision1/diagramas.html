Introducción al diseño de diagramas E/R

Qué veremos: conceptos básicos, elementos del modelo, pasos para diseñar y un ejemplo práctico.

Objetivos de la sesión

  • Comprender qué es un diagrama Entidad–Relación (E/R).
  • Identificar entidades, atributos y relaciones en un problema real.
  • Practicar la transformación de requisitos a un diagrama E/R.

¿Por qué modelar con E/R?

  • Permite representar la estructura de datos antes de implementarla.
  • Facilita comunicación entre analistas, diseñadores y desarrolladores.
  • Reduce ambigüedades y evita redundancias en la base de datos.

Elementos básicos: Entidad

Entidad: objeto o concepto del mundo real con existencia independiente (ej. Usuario, Producto).

Representación típica: rectángulo con el nombre de la entidad.

Elementos básicos: Atributo

Atributo: característica o propiedad de una entidad (ej. nombre, fecha_nacimiento).

  • Atributo simple vs compuesto
  • Atributo multivaluado
  • Atributo derivado

Elementos básicos: Relación

Relación: asociación entre dos o más entidades (ej. Usuario — realiza — Pedido).

Representación típica: rombo o línea conectando entidades con nombre de la relación.

Cardinalidad y participación

  • Cardinalidad: indica cuántas instancias de una entidad se asocian con otra (1:1, 1:N, N:M).
  • Participación: opcional o total (si toda instancia debe participar en la relación).

Reglas prácticas y consejos

  • Identificar primero los sustantivos (posibles entidades) y verbos (posibles relaciones) del enunciado.
  • Evitar atributos repetidos; si aparecen, considerar una nueva entidad.
  • Modelar relaciones N:M con una entidad intermedia (tabla puente) en diseño relacional.
  • Marcar claramente claves primarias y atributos obligatorios.

Pasos para crear un diagrama E/R

  1. Recopilar requisitos y leer el enunciado o caso de uso.
  2. Listar posibles entidades y atributos.
  3. Determinar relaciones entre entidades y cardinalidades.
  4. Refinar atributos (simples, compuestos, multivaluados).
  5. Decidir claves primarias y restricciones de integridad.
  6. Revisar con stakeholders y normalizar si es necesario.
  7. Preparar la migración a modelo relacional (tablas y claves foráneas).

Ejemplo práctico (resumen)

Caso: Biblioteca

  • Entidades: Libro, Autor, Usuario, Préstamo.
  • Relaciones: Autor escribe Libro (1:N), Usuario realiza Préstamo (1:N), Préstamo involucra Libro (N:M → entidad intermedia: DetallePréstamo).

Actividad práctica

Entregar en 30–45 min: diseñar el diagrama E/R para un sistema de reservas de salas de reuniones.

Pistas: identificar entidades (Sala, Reserva, Usuario), atributos clave y cardinalidades.

Recursos y lectura recomendada

  • Guías de modelado E/R (libros y artículos básicos).
  • Herramientas: draw.io, dbdiagram.io, Lucidchart.
  • Práctica adicional: convertir el diagrama a tablas y aplicar normalización.

Resumen

El diagrama E/R es la base para diseñar bases de datos bien estructuradas: identifica entidades, atributos y relaciones antes de escribir SQL.
